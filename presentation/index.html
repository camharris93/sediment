The problem

Two instincts that usually cancel out.

move fast

Analysts want answers now. Ad-hoc SQL, notebooks, one-off extracts. You get speed — and nothing reproducible, nothing tested, numbers no one can trace back to a query.

be rigorous

Engineers want reproducible, tested, version-controlled pipelines. You get trust — and a slow path to onboard new data, with every stakeholder waiting in the queue.

And now there's AI in the middle — fast, fluent, and confidently wrong. Most teams either keep it out entirely, or let it generate pipeline code nobody fully trusts. sediment refuses that trade.

The one decision that matters
Deterministic core.
AI at the edges.
The raw → staging → marts flow is hand-written, tested dbt SQL. It produces the same result on every run, on every clone. AI helps build it, operate it, and query it — but never executes the transforms.
Reproducibility is the product.  Automate what's safe; curate what matters.

The name is the architecture

Data settles into layers.

Like sediment: loose material at the bottom, compacted in the middle, crystallized into something durable on top.

martshuman-curated
The semantics — what business questions the data answers. Hand-written, tested dbt. No tool can know what matters in your data, so this stays a human judgment call.
staginggenerated · reviewed
A clean, typed, tested per-row contract. snake_cased, cast, junk filtered. AI proposes it from a profile; you review and commit.
rawmechanical
Every source column, landed untouched and auto-typed. No renames, no casts — that's staging's job. Faithful to the source.
⬢ One local warehouse.duckdb file holds it all — many datasets coexist, each in its own schemas.

AI, at the seams

Three places AI earns its keep.

build

Scaffolding copilot. Profiles a new source and proposes a typed staging model, conservative tests, and mart stubs — for you to review and commit. A deterministic baseline runs with no key at all.

operate

Orchestration brain. Drives load → dbt run → test, explains failures in plain English with the offending rows, and flags row-count drift vs. the last run. The pipeline still fails loudly without a key.

consume

NL→SQL agent. A plain-English question, grounded in the live schema, becomes read-only SQL over your curated marts — where every number traces back to the query that produced it.

Trust, by construction

A question runs a gauntlet before it touches a row.

Seven layers. The model proposes; the framework verifies — against the real schema, not the model's word for it.

L1
Intent

L3
Static check

the point

A query that can't be validated is a transparent failure — not a confident guess. L3/L4 violations feed back to L2 in a bounded self-correction loop. If it still can't be proven safe, it says so.

From question to infrastructure

A good answer shouldn't evaporate.

Promote a validated chat answer into a reviewable dbt model — preview it in an isolated sandbox, save it as models/marts/<name>.sql, or build it straight into marts. The unit of work is always reviewable dbt SQL; the chat never silently mutates the curated marts.

the lifecycle

An ad-hoc question becomes durable, tested infrastructure in a couple of clicks — instead of a forgotten cell in someone's notebook.

governance · view vs build

Sharing a report never hands out the ability to build models. Build is a property of the deployment, not a per-user toggle — and it's refused server-side, not just hidden in the UI.

Why engineers can trust it

The framework is tested — not just the data.

112
Tests, offline & key-free
The deterministic engine under test — including adversarial tests that the read-only guard refuses local-file reads, file writes, and multi-statement payloads.
RO
Read-only by construction
The NL→SQL agent runs against the warehouse read-only. It can't write. It can't read your filesystem. The guard is the security boundary — and it's covered by the attacks it defends against.
CI
Reproducibility, proven on every push
CI builds a fresh clone into a green, tested pipeline on Linux & Windows × Python 3.10 / 3.13 — fully offline.
=
Same input, same marts
No LLM in the transform path means the output is a pure function of the source and the committed SQL. Every clone, every run.
What this unlocks

Where it fits in a real team.

onboarding

Land a new source — a vendor export, an ops dump, a teammate's CSV — into a typed, tested staging layer in minutes, not a sprint.

self-serve

Give stakeholders a governed, plain-English interface over curated marts — with answers that trace to SQL, and no path to mutate anything.

prototype → prod

Turn a chat question into reviewed dbt — a real model in version control, not a one-off query lost in a thread.

a reusable stance

“AI at the edges, never in the critical path” is a pattern you can carry into far bigger pipelines than this one.

If you remember three things

The whole idea, in three lines.

01
Determinism is non-negotiable.
AI lives at the edges — build, operate, consume — and never in the transform that has to be reproducible.
02
Trust is engineered, layer by layer.
A natural-language answer is verified against the real schema before it's believed — or it fails transparently.
03
Questions become infrastructure.
The ad-hoc → tested-dbt path is one motion, with governance built into the deployment.
